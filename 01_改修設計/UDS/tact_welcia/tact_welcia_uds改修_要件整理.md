# TACT ウエルシア案件 UDS改修 要件整理

## 概要

ウエルシア顧客の交換依頼に限り、通常の支店キッティング・発送ではなく、TACT（受付窓口）＋オムロンを経由した対応を行う。

---

## 現状の仕組み

- 顧客住所によってキッティング作業を担当する支店が NeOS/UNIS 側で決定される
- `import_contracts` によって UDS に「この顧客はどの支店か」が連携される
- 支店がキッティング・発送を担当する

---

## 依頼種別ごとの対応方針

| 依頼種別 | 対応 |
|---------|------|
| 通常新設依頼 | 変更なし。import_contracts で受け取った支店でキッティング・発送 |
| **交換依頼（ウエルシアのみ）** | TACT＋オムロンを利用。オムロンから顧客へ直送 |

---

## 交換依頼の運用フロー

```
① TACT が電話を受け付ける
② TACT が UDS を操作して修理交換ボタンを押下する
③ TACT が新しいシリアルを交換依頼に引き当てる
④ 交換品はオムロンから顧客のもとへ設置しに行く。
```

- リンク品の実物はオムロンが保持（通常は支店が保持）
- オムロンは全国約140拠点。顧客住所ごとに担当拠点を決める必要がある

---

## 実装方針：案1（オムロン拠点を現状にUDS倉庫と同様に設定し既存フローに乗せる）

### 基本方針

オムロン各拠点を UDS 上の「倉庫（branch）」として `branch_replacement` に登録し、交換依頼時に既存の修理交換フローをそのまま実行する。

### フロー

```
修理交換ボタン押下
 ↓
UDS がウエルシア顧客判定（※識別キーはMTGで確認待ち）
 ↓ ウエルシアと判定された場合
担当オムロン拠点を決定（下記 案1-A / 案1-B のどちらか）
 ↓
branch_replacement 上のオムロン拠点CDをキッティング担当にセット
 ↓
既存の修理交換フロー（シリアル引き当て → 発送指示）を実行
```

### オムロン支店の振り分け方法（MTGで方針を決定する）

#### 案1-A：マッピングテーブルで自動振り分け

- 新テーブル「顧客住所 → オムロン拠点CD」マッピングを作成
- 修理交換ボタン押下時にテーブルを参照してオムロン拠点を自動決定
- マッピングのメンテナンスは通信事業部がCSVアップロードで行う

| 観点 | 評価 |
|------|------|
| TACT操作コスト | ◎ 自動なので選択不要 |
| 開発コスト | 高（マッピングテーブル＋CSVアップロード画面が必要） |
| メンテナンス | 通信事業部がCSVで管理 |
| 誤振り分けリスク | マッピングデータが古い・誤っている場合に気づきにくい |

#### 案1-B：TACT がプルダウンでオムロン支店を手動選択

- 修理交換ボタン押下時にオムロン拠点一覧をプルダウン表示
- TACT が該当オムロン支店を選択して進む
- マッピングテーブル・CSVアップロード画面は不要

| 観点 | 評価 |
|------|------|
| TACT操作コスト | △ 毎回選択が必要 |
| 開発コスト | 低（マッピングテーブル・アップロード画面が不要） |
| メンテナンス | オムロン拠点マスタのみ管理すればよい |
| 誤振り分けリスク | TACT側のオペレーションミスは起こりうる |

#### 案1-A / 案1-B 比較

| 観点 | 案1-A（自動） | 案1-B（手動選択） |
|------|-------------|----------------|
| 開発コスト | 高 | 低 |
| TACT操作 | 楽 | 手間あり |
| 初期データ整備 | マッピング140件分必要 | オムロン拠点マスタのみ |
| 誤振り分けリスク | データ起因で気づきにくい | 画面で確認できる |
| 将来の拡張性 | マッピング更新で対応可 | プルダウン選択肢の追加のみ |

## 必要なDB変更

### 新テーブル①：オムロン拠点マスタ（共通）

```sql
CREATE TABLE omron_branches (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  branch_cd   VARCHAR(13) COMMENT 'オムロン拠点CD（branch_replacementのbranch_cdと紐付け）',
  branch_name VARCHAR(40),
  zip_cd      VARCHAR(8),
  address1    VARCHAR(256),
  address2    VARCHAR(256),
  created_at  DATETIME,
  updated_at  DATETIME,
  deleted_at  DATETIME
);
```

### 新テーブル②：顧客住所 → オムロン拠点マッピング（案1-Aのみ）

```sql
CREATE TABLE welcia_area_omron_mapping (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  pref_cd         VARCHAR(2)  COMMENT '都道府県CD or 郵便番号エリア（粒度はMTGで確認）',
  omron_branch_cd VARCHAR(13) COMMENT 'omron_branches.branch_cd',
  created_at      DATETIME,
  updated_at      DATETIME,
  deleted_at      DATETIME
);
```

### 既存テーブルへの変更

| テーブル | 変更内容 |
|---------|---------|
| `branch_replacement` | オムロン140拠点を新規レコードとして登録（スキーマ変更なし、データ投入のみ） |

## 現状決まっていること

| # | 内容 |
|---|------|
| 対象顧客 | ウエルシアのみ |
| 新設依頼 | 既存フローのまま |
| 交換依頼の運用フロー | TACT受付 → UDS操作 → シリアル引き当て → オムロンから設置 |
| 在庫保持 | リンク品の実物はオムロンが保持 |
| オムロン拠点数 | 全国約140拠点 |
| 実装方針 | 案1（UDS倉庫と同様に設定し既存フローに乗せる） |
| スキーマ変更 | `branch_replacement` はデータ投入のみ（スキーマ変更なし） |
| 新テーブル | オムロン拠点マスタは確定。マッピングテーブルは案1-A採用時のみ |
| マッピングメンテナンス担当 | 通信事業部（CSVアップロード画面が必要） |

## MTGで確認待ちの事項

| # | 論点 | 確認内容 |
|---|------|---------|
| ① | ウエルシア顧客の識別キー | import_contracts で受け取るデータ内に識別できるキーがあるか |
| ③ | TACTのUDS権限 | 既存の支店権限を流用するか、TACT専用ロールを新規作成するか |
| ④ | オムロン在庫のシリアル登録 | UDSに登録するか否か。登録する場合は誰がいつ登録するか |
| ⑤ | オムロンへの発送指示方法 | 帳票・メール・APIのどれか |
| ⑥ | 交換依頼のステータス管理 | UDSで管理するステータスの種類と遷移 |
| 振り分け方法 | 案1-A（自動マッピング）か案1-B（TACTがプルダウン手動選択）か | 通信事業部の意向次第 |
| 案1-Aの場合 | マッピングの粒度 | 都道府県単位か郵便番号単位か |
